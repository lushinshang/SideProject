L121 No-code / Low-code 概念

No-code vs Low-code 差異比較

比較維度 No-code Low-code
目標用戶 非技術背景使用者（市民開發者） 具有基本技術背景的開發者
開發方式 純視覺化拖放介面，零程式碼 視覺化 + 少量程式碼客製化
適用場景 快速原型、小型應用、簡易工作流 複雜邏輯、企業級應用、高彈性需求
擴展性 有限，自訂程度低 高，可整合複雜系統（CRM/ERP）
典型用途 表單、簡易工作流、行銷頁面 中大型企業內部系統、API整合

⚠️ 考試易錯：No-code「完全不需要程式碼」≠「完全不需要任何技術知識」；Low-code「少量程式碼」≠「只適合專業開發者」

優勢與限制

優勢（三大核心）： 1. 降低開發成本：減少對專業程式設計師依賴 2. 縮短上市時間：快速原型→快速部署 3. 加速數位轉型：促進IT與業務部門協作

限制： - 無法完全取代技術人員（複雜/高度客製化場景仍需專業人員） - 過度依賴可能導致使用者缺乏對AI的深層理解 - 可能引發模型偏差或誤用風險

選擇平台的6大關鍵因素

  1. 目標用戶與技術需求
  2. 功能與擴展性（系統整合能力、高度自訂性）
  3. 安全性與合規性（GDPR、CCPA）
  4. 成本與效益（TCO總擁有成本、ROI投資回報率）
  5. 技術支援與社群資源
  6. 市場評價與案例

整合方式 說明

個人化App快速開發 用戶數據分析→個人化應用

AI民主化（AI Democratization）

  • 將AI技術從少數專家擴展至更廣泛社會
  • No-code/Low-code是實現AI民主化的重要工具
  • 市民開發者（Citizen Developer）：非技術背景人員利用這些工具自動化日常工作

挑戰與解方

挑戰 解方
模型準確性/可靠性 建立多層次驗證機制+人工審核
資料隱私/安全 加密、匿名化、權限控管
道德/倫理風險 倫理準則+公平性檢測工具
技術整合/平台適配 開放式API平台+雲端運算

L122 生成式AI應用領域與工具使用

生成式AI基本概念

  • 核心特徵：透過模型生成新內容，而非僅分析或辨識現有數據
  • 關鍵技術架構：
  • GAN（生成對抗網路）：高品質圖像生成
  • VAE（變分自編碼器）：穩定性與機率生成
  • Transformer + Self-Attention：GPT系列，語言生成革命
  • RLHF（人類回饋強化學習）：使生成結果更符合用戶需求
  • ChatGPT於2022年11月正式推出，開啟生成式AI革命

模態 代表工具 應用場景
文本/語言 ChatGPT、Claude、Gemini 摘要、翻譯、客服、程式碼生成
圖像 DALL-E、Midjourney、Stable Diffusion 數位藝術、設計、廣告
聲音/語音 Whisper（語音辨識）、Suno（音樂生成） 語音轉文字、音樂創作

🔑 核心概念：提示工程的目的是優化提示以提升AI輸出品質，透過精確的提示詞讓AI輸出更符合需求

生成式AI常見挑戰

  • AI幻覺（Hallucination）：遇到不懂問題時，AI傾向「一本正經地胡說八道」
  • 知識截止日期：訓練資料有截止日期，無法獲取最新資訊
  • 偏見（Bias）：訓練數據若存在偏見，將影響輸出公正性
  • 資安隱憂：商業機密不能直接輸入公有雲模型

RAG（檢索增強生成）

為什麼需要RAG？

通用LLM的問題：
知識限制 → 無貴公司SOP/技術手冊
幻覺問題 → 不懂時仍會捏造答案
資安隱憂 → 商業機密不能上傳

RAG標準運作流程

用戶提問
→ [檢索層] 系統在企業知識庫搜尋相關文件（向量語意搜尋）
→ [組合層] 問題 + 檢索到的資料 → 組合成增強提示詞
→ [生成層] LLM根據提供資料生成答案（可溯源、減少幻覺）

Prompt vs RAG vs Fine-tuning 三角選擇

比較維度 Prompt工程 RAG Fine-tuning（微調）
直觀比喻 給AI看臨時小抄 讓AI查閱圖書館 送AI去補習深造
原理 在對話框貼上背景資訊 系統先去資料庫撈資料再餵給AI 修改模型神經網路參數（改變大腦結構）
知識時效性 依賴使用者當下提供 即時（Real-time） 滯後（Static），需重新訓練才能更新
幻覺風險 視上下文品質而定 低（可溯源引用來源） 中，仍可能幻覺且難查證
建置成本 ⭐極低 ⭐⭐中等（需向量資料庫+檢索系統） ⭐⭐⭐高（算力貴、資料清洗耗時）
適用場景 單次短篇翻譯/摘要 企業知識庫、FAQ、法規查詢 學習特殊語氣、格式、冷門專業術語

✅ 結論：企業知識庫的首選通常是 RAG

L123 生成式AI導入評估規劃

企業導入AI四大發展階段

構想 → 設計 → 驗證POC → 實施/營運
階段 目的 主要工作
1. 構想 挑選合適AI方案 掌握企業課題→盤點資源→方案排序
2. 設計 確認AI需求規格 導入目的→數據確認→應用情境→評估成本
3. 驗證POC 驗證AI效果 模型建置→運作模式→評估ROI
4. 實施/營運 持續發揮AI價值 模型監控→應用擴散

構想階段的三個步驟

  1. 步驟一：掌握企業課題
  2. 經營目標明確化 → 選定應用範圍 → 提取改善項目 → AI必要性
  3. 步驟二：檢視AI方案與企業資源
  4. 盤點AI相關方案（硬體+軟體+演算法）
  5. 檢視企業資源（數位化程度、數據、人力、時間、預算）
  6. 步驟三：確定應用領域優先順序
  7. 評估導入容易程度 + 導入可能好處 → 排序

導入評估五大標準

  1. 需求與現狀評估：痛點識別、應用場景分析、技術對接性
  2. 資源與基礎設施評估：技術人才、數據品質、硬體架構
  3. 分階段策略：試點→階段性擴展→長期目標→持續優化
  4. 員工技能與文化導入：技術培訓、實務操作、數位轉型文化
  5. 風險評估與管理：資料安全、倫理法規、技術風險、資源風險

❗ 企業導入生成式AI的第一步：設定可量化、可追蹤的經營目標（非購買軟體或硬體升級）

三大成功關鍵因素： 1. 清晰定義痛點 2. 技術匹配 3. 培養人才

企業規模導入階段

階段 重點
初始（試點驗證） 小範圍試點、低成本驗證可行性、收集回饋
成長（技術推廣整合） 跨部門協作、從單點擴展至多個業務環節
成熟（全面整合創新） 優化精進、探索創新應用、差異化競爭

ROI計算與評估

  • ROI公式：（收益 - 成本）/ 成本 × 100%
  • NPV（淨現值）：考慮資金時間價值
  • IRR（內部報酬率）：評估長期財務可行性
  • 敏感性分析：模擬市場條件變化對方案收益的影響

模型驗證指標

  • BLEU / ROUGE / Perplexity：文本生成品質評估
  • Accuracy、Recall、Error Rate：模型效能追蹤
  • Data Drift（數據漂移）：輸入數據分佈變化監控

生成式AI風險管理

主要風險類型

風險類型 說明
資料隱私與安全 訓練數據洩漏、反向工程、提示詞攻擊、對抗性攻擊
倫理偏見 訓練數據偏見→性別、種族歧視（如Amazon招聘AI案例）
法律合規 GDPR、CCPA等隱私法規；版權問題
AI幻覺 模型捏造事實，特別是高風險場景（醫療、金融）

六大風險應對策略

策略 說明 考試記憶
風險溯源 審核數據來源合法性與可靠性 追根溯源
風險文化 員工培訓、定期教育 建立文化
風險接受 影響有限時接受風險，制定應急計畫 理性選擇
風險緩解 數據管理、內容審查、合規政策 減少傷害
風險迴避 暫緩高風險應用 不做
風險轉移 外包給第三方、購買保險 外包=風險轉移

⚠️ 考試考點：「企業將資料安全管理外包給第三方」= 風險轉移

防範措施

  • 差分隱私（Differential Privacy）：加入特定噪聲，防止敏感資料外洩（隱私保護黃金標準）
  • 數據匿名化/假名化：GDPR合規要求
  • 身份驗證與授權機制：嚴格控管存取
  • 內容篩選：過濾有害輸出（如Azure OpenAI內容篩選）

重要法規

  • GDPR（歐盟一般資料保護規則）：個人資料保護
  • CCPA（加州消費者隱私法案）：美國地區資料保護

考試新增概念（超出學習指引，114/4考古題分析）

MCP架構（Model Context Protocol）

定義：Anthropic主導，標準化AI智能體連接外部數據、工具與APIs的通用接口標準

架構流程：

AI Host（如Claude Desktop）
→ MCP Client（負責連接）
→ MCP Server（特定服務/工具提供者）
→ 資料查詢（SQL DB、GitHub、Slack等）
→ 回傳結果給AI Host

核心價值：解決智能體與不同軟體生態互通的碎片化難題，讓AI智能體無縫連接企業內部各種數據孤島，無需為每個平台編寫特定連接器

A2A架構（Agent-to-Agent）

定義：規範不同廠商AI智能體之間通訊、協作與任務交付的協議

運作方式：

Client Agent（發起方，如OpenAI Agent）
→ 身份驗證 + 任務委託
→ Remote Agent（執行方，如Anthropic Agent）
→ 完成專職任務後回傳結果

核心價值：建立「智能體互聯網」，讓不同廠商的專家模型如微服務般組成高效團隊

Agentic AI（智能體AI）

演進路徑：

聊天機器人（對話）→ AI助理（輔助）→ Agentic AI（自主執行多步任務）

核心特性： - 環境感知（接收部分觀測數據） - 維護內部狀態（跨對話記憶） - 自主調用工具 - 多步規劃與執行 - 自我修正能力

比較 Agentic Workflow（工作流） Autonomous Agent（自主智能體）
路徑 預定義結構化路徑（確定性） 動態生成路徑（隨機性）
優勢 高可預測性、可控性、安全性 高靈活性、解決開放式問題
風險 低，僅限節點失效 高，可能幻覺或不可預知連鎖反應

Solution Graph（解決方案圖）搜尋策略：在Agentic AI中，模型透過搜尋可能的解決方案路徑（Solution Graph）來找到最優解，而非線性執行

有界自主（Bounded Autonomy）： - 為自主AI智能體設定操作界限與安全護欄 - 三級分類：低風險（全自動化）、中風險（受監控自主）、高風險（人類主導）

Context Engineering（上下文工程）

定義：優化輸入AI的上下文（Context）資訊的系統性方法，目的是最大化AI輸出品質

與Prompt Engineering的關係： - Prompt Engineering：聚焦單一提示詞設計 - Context Engineering：更全面，包含系統設計、記憶管理、工具調用、知識注入等整個上下文環境的工程化優化

核心目的： 1. 減少AI幻覺（Hallucination） 2. 提升答案準確度與相關性 3. 控制模型行為範圍（Grounding） 4. 最大化有效Token利用率

APE（Automatic Prompt Engineer）

定義：自動化提示詞工程，利用AI自動生成、評估、優化提示詞，而非人工手寫

運作方式：

目標任務描述
→ APE系統自動生成多個候選提示詞
→ 自動評估各提示詞的輸出品質
→ 選出最優提示詞（可迭代優化）

意義：將提示工程從「藝術」轉為「科學」，降低對專家人工設計Prompt的依賴

聯邦學習（Federated Learning）用於隱私保護

定義：在不移動本地數據的前提下，透過交換模型參數來共同優化全局模型的框架

核心優勢： - 解決醫療、金融等敏感數據無法跨機構整合的難題 - 數據「不出門」，只傳遞模型更新（梯度/參數）

運作流程：

各機構本地訓練模型（數據不離開）
→ 上傳模型參數（非原始數據）至中央伺服器
→ 中央伺服器聚合參數→更新全局模型
→ 下發更新後的全局模型給各機構

應用場景： - 醫院間的疾病診斷模型（病歷不出院） - 銀行間的詐騙偵測（交易數據不共享） - 手機端個性化模型（個人資料留在設備）

容易混淆的概念

1. RAG vs Fine-tuning vs Prompt Engineering

記憶口訣：
Prompt = 臨時小抄（最快、最便宜、不持久）
RAG = 查圖書館（即時、可溯源、企業首選）
Fine-tuning = 去補習班（改造大腦、貴、學口吻不學知識）

2. No-code vs Low-code

No-code = 完全不寫程式 = 非技術人員 = 簡單應用
Low-code = 少量程式碼 = 技術人員 = 複雜/企業級應用
（兩者都不能完全取代專業開發者）

3. 風險轉移 vs 風險緩解

風險轉移 = 外包/買保險 = 責任轉給第三方（但仍需監控）
風險緩解 = 採取措施降低風險發生機率或影響
風險迴避 = 不做這件事
風險接受 = 知道有風險但仍繼續（制定應急計畫）

4. MCP vs A2A

MCP = AI智能體 ↔ 外部工具/資料庫（工具整合協議）
A2A = AI智能體 ↔ AI智能體（跨廠商智能體通訊協議）

5. Agentic Workflow vs Autonomous Agent

Agentic Workflow = 預定義流程 = 可控、可預測（企業生產環境推薦）
Autonomous Agent = 自主規劃 = 靈活但不可控（研究/探索任務）

6. 傳統AI vs LLM

比較項目 傳統AI (ML/NN) LLM
資料需求 大量標註資料（數千/萬筆Label） Zero-shot 或 Few-shot
適應新任務 需重新蒐集資料、重新訓練 修改Prompt即可
啟動成本 高（標註昂貴、訓練時間長） 極低（寫好Prompt即可測試）

7. GAN vs VAE vs Transformer（生成式AI基礎）

GAN（生成對抗網路）= 圖像生成主力 = 生成式AI的基礎技術（考試答案）
VAE（變分自編碼器）= 穩定的機率生成
Transformer = GPT/語言模型基礎 = 多模態生成（DALL-E、Stable Diffusion）
