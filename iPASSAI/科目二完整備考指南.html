L121 No-code / Low-code 概念

No-code vs Low-code 差異比較 ⭐

比較維度 No-code Low-code
目標用戶 非技術背景使用者（市民開發者） 具有基本技術背景的開發者
開發方式 純視覺化拖放介面，零程式碼 視覺化 + 少量程式碼客製化
適用場景 快速原型、小型應用、簡易工作流 複雜邏輯、企業級應用、高彈性需求
擴展性 有限，自訂程度低 高，可整合複雜系統（CRM/ERP）
典型用途 表單、簡易工作流、行銷頁面 中大型企業內部系統、API整合

⚠️ 考試易錯：No-code「完全不需要程式碼」≠「完全不需要任何技術知識」

優勢與限制

三大核心優勢：

  1. 降低開發成本：減少對專業程式設計師依賴
  2. 縮短上市時間：快速原型→快速部署
  3. 加速數位轉型：促進IT與業務部門協作

限制：

  • 無法完全取代技術人員（複雜/高度客製化場景仍需專業人員）
  • 過度依賴可能導致使用者缺乏對AI的深層理解
  • 可能引發模型偏差或誤用風險

考題答題策略（來自114/4實際考題）

Q1：Low-code中「模型（Model）」的正確理解

答案：模型是抽象描述資料結構、業務流程與介面邏輯的核心元素

❌ 錯誤：不是「被自動程式碼取代」，不是「只用UML」

Q6：No-code/Low-code與生成式AI整合策略

答案：透過生成式AI在No-code平台中自動建立介面模板，結合使用者數據即時生成個人化功能

❌ 錯誤：「完全依賴AI不經人工驗證」是錯誤選項

Q33：No-code解決MLOps挑戰

答案：透過可視化建模介面，降低技術門檻並促進跨部門協作

選擇平台的6大關鍵因素

  1. 目標用戶與技術需求
  2. 功能與擴展性（系統整合能力、高度自訂性）
  3. 安全性與合規性（GDPR、CCPA）
  4. 成本與效益（TCO總擁有成本、ROI投資回報率）
  5. 技術支援與社群資源
  6. 市場評價與案例

AI民主化（AI Democratization）

  • 將AI技術從少數專家擴展至更廣泛社會
  • No-code/Low-code是實現AI民主化的重要工具
  • 市民開發者（Citizen Developer）：非技術背景人員利用工具自動化日常工作

L122 生成式AI應用領域與工具使用

生成式AI基本概念

  • 核心特徵：透過模型生成新內容，而非僅分析或辨識現有數據
  • 關鍵技術架構：
  • GAN（生成對抗網路）：高品質圖像生成，生成式AI的基礎技術（考試標準答案）
  • VAE（變分自編碼器）：穩定性與機率生成
  • Transformer + Self-Attention：GPT系列，語言生成革命
  • RLHF（人類回饋強化學習）：使生成結果更符合用戶需求
  • ChatGPT於2022年11月正式推出，開啟生成式AI革命

主要應用領域 ⭐

模態 代表工具 應用場景
文本/語言 ChatGPT、Claude、Gemini 摘要、翻譯、客服、程式碼生成
圖像 DALL-E、Midjourney、Stable Diffusion 數位藝術、設計、廣告
聲音/語音 Whisper、Suno 語音轉文字、音樂創作
影片 Sora、Veo 影片生成、特效、劇本視覺化
程式碼 GitHub Copilot、Cursor 程式碼補全、自動生成、重構

GitHub Copilot考題重點 ⭐

正確描述（114/4 Q13）：由OpenAI的Codex模型提供技術支援，可即時在開發者編輯程式碼時給出整行或整個函式建議

❌ 錯誤陷阱：不是「程式碼片段查詢工具」；不是「僅適用開源專案」；不是「靜態分析技術」

Prompt工程類型詳解 ⭐⭐

技術 說明 最適用情境
Zero-shot 不提供任何範例，直接提問 情感分析、翻譯等通用任務
One-shot 提供1個範例 需要特定格式輸出
Few-shot 提供2-5個範例 需要特定風格/格式
Chain-of-Thought (CoT) 要求AI逐步推理，線性流程 退款查詢等單一線性流程
Tree of Thoughts (ToT) 探索多條推理路徑，比較方案 活動規劃等需比較多方案
Graph Prompting 處理非線性關係/網絡結構資訊 社交關係分析、跨國合規對比
Role Prompting 指定AI扮演特定角色 專業領域回覆
System Prompt 設定模型行為的系統級指令 定義AI助理人設、範圍限制

🔑 CoT vs ToT考試判斷口訣：

  • 查詢退款流程（單一線性）→ CoT
  • 規劃多方案活動（多分支比較）→ ToT

Zero-shot失敗情境（Q23）：最可能失敗：從表格中擷取所有城市的最高氣溫（結構化數據提取，需要示範格式）

APE超長上下文限制（Q5）：

答案：回饋機制通常僅針對局部片段，難以全面評估最終輸出品質

Graph Prompting挑戰（Q4）：

答案：圖結構轉換為文字提示時，可能導致部分關聯資訊遺失

生成溫度（Temperature）與風格控制

Q26：確保商品描述風格一致

答案：降低生成溫度，以減少隨機性並提升風格一致性

Q15：動態調整回覆風格

答案：利用控制變量（Control Tokens）或風格標籤在同一模型內動態調整

RAG（檢索增強生成）⭐⭐

為什麼需要RAG？

通用LLM的問題：
  知識限制 → 無法得知最新/企業內部知識
  幻覺問題 → 不懂時仍會捏造答案
  資安隱憂 → 商業機密不能上傳公有雲

RAG標準運作流程

用戶提問
  → [檢索層] 系統在企業知識庫搜尋相關文件（向量語意搜尋）
  → [組合層] 問題 + 檢索到的資料 → 組合成增強提示詞
  → [生成層] LLM根據提供資料生成答案（可溯源、減少幻覺）

Prompt vs RAG vs Fine-tuning 三角選擇 ⭐⭐

比較維度 Prompt工程 RAG Fine-tuning（微調）
直觀比喻 給AI看臨時小抄 讓AI查閱圖書館 送AI去補習深造
知識時效性 依賴使用者當下提供 即時（Real-time） 滯後（Static）
幻覺風險 視上下文品質 低（可溯源） 中
建置成本 ⭐極低 ⭐⭐中等 ⭐⭐⭐高
適用場景 單次翻譯/摘要 企業知識庫、FAQ 學習特殊語氣/格式

Fine-tuning + RAG混合策略（Q17）：

需要語氣一致+即時查詢新文件+避免頻繁重訓

最佳答案：保留語氣相關Fine-tuning，僅透過RAG更新文件內容

記憶：「能力靠Fine-tuning，知識靠RAG」

RAG品質提升策略（Q18、Q47、Q48）：

  • Q18：回覆品質波動→先優化檢索系統品質（確保取得最新文件）
  • Q47：檢索結果不相關→Chunking（切分）→提高檢索相關性、降低長上下文噪音
  • Q48：多向量檢索器→同時處理多種資訊表示，提升跨文本型態檢索效果

知識蒸餾（Knowledge Distillation）⭐

定義：讓小型模型學習大型模型的行為，在降低運算成本的同時維持效能

Q42答案：RAG系統運算成本過高→引入知識蒸餾→小模型學大模型知識，降低成本同時維持品質

生成式AI常見挑戰

  • AI幻覺（Hallucination）：矛盾輸入→隨機採信其中一方（Q10答案：可能生成幻覺）
  • 知識截止日期：需RAG即時更新
  • 偏見（Bias）：訓練數據偏見→公平性檢測（Q38答案）
  • 資安隱憂：本地部署LLM的核心優勢（Q12答案：敏感資料不傳第三方）

模型評估基準測試（Benchmark）⭐

資料集 用途
MMLU 多領域、多任務語言理解（人文、科學、社會科學）
GSM8K 數學推理（小學數學題）
MATH 進階數學推理
C-Eval 中文專業知識

Q45答案：問「多領域多任務語言理解，涵蓋人文科學社會科學」→ MMLU（不是C-Eval）

Q43模型規模影響：模型越大，Benchmark結果可能提升，但幅度取決於訓練數據品質與資源配置

L123 生成式AI導入評估規劃

企業導入AI四大發展階段

階段 目的 主要工作
1. 構想 挑選合適AI方案 掌握企業課題→盤點資源→方案排序
2. 設計 確認AI需求規格 導入目的→數據確認→應用情境→評估成本
3. 驗證POC 驗證AI效果 模型建置→運作模式→評估ROI
4. 實施/營運 持續發揮AI價值 模型監控→應用擴散

❗ 企業導入AI的第一步：明確定義目標並設定KPI（非購買軟體或硬體升級）

Q30正確順序（物流公司導入AI）：

2（設定目標+KPI）→ 3（蒐集清理資料）→ 4（選技術供應商）→ 1（建系統邏輯）

三大成功關鍵因素：清晰定義痛點 + 技術匹配 + 培養人才

資料安全與合規策略（Q24、Q31）

Q24（保險公司資料安全優先）：

最符合：納入法遵稽核單位，設定KPI，透過MVP驗證成效

Q31（電商個資保護）：

最佳措施：資料最小化與去識別化，確保AI訓練與生成中不直接處理敏感個資

風險管理策略 ⭐⭐

策略 說明 考試記憶
風險溯源 審核數據來源合法性 追根溯源
風險文化 員工培訓、定期教育 建立文化
風險接受 影響有限時接受，制定應急計畫 理性選擇
風險緩解 數據管理、內容審查、合規政策 減少傷害
風險迴避 暫緩高風險應用 不做
風險轉移 外包給第三方、購買保險 外包=風險轉移

Guardrails防護機制（Q46）⭐

Guardrails的主要目的：

  • ✅ 檢查輸入避免危險需求
  • ✅ 過濾驗證AI輸出符合安全標準
  • ✅ 確保符合法規與產業安全規範
  • ❌ 不是「重建追蹤AI模型全部推理過程」

Prompt Injection攻擊防護（Q36）⭐

最佳策略：導入輸入檢測與回應審核流程，防止敏感指令被執行

記憶：雙層防護（輸入檢測+回應審核）

ROI計算與評估

  • ROI公式：（收益 - 成本）/ 成本 × 100%
  • NPV（淨現值）：考慮資金時間價值
  • IRR（內部報酬率）：評估長期財務可行性

模型驗證指標

  • BLEU / ROUGE / Perplexity：文本生成品質評估
  • Accuracy（準確率）、Recall（召回率）、Error Rate：模型效能追蹤
  • Data Drift（數據漂移）：輸入數據分佈變化監控

Data Drift定義（Q37）：

訓練時使用的資料分佈，與部署後實際輸入資料的統計特徵隨時間逐漸出現差異，導致模型表現衰退

延遲測試指標（Q35）：

客戶從輸入問題到收到第一個完整回應所需的時間（不是每分鐘回覆數量）

🆕 MCP架構（Model Context Protocol）⭐⭐⭐⭐⭐

核心定義

  • Anthropic主導的開放協議，讓LLM能標準化地與外部工具/資源對接
  • 被稱為「AI的USB-C接口」，解決AI與不同生態互通的碎片化難題

三大角色與架構流程（Q7 必考）⭐⭐⭐⭐⭐

正確流程：
AI Host → MCP Client → MCP Server → 資料查詢 → 結果回傳 AI Host
角色 職責
AI Host 宿主（如Claude Desktop、VS Code插件），用戶端與LLM的橋樑
MCP Client 協議連接器，存在於Host內部，負責與Server維護連線
MCP Server 單職責服務，公開三大原語：Resources（資源）、Tools（工具）、Prompts（提示模板）

⚠️ 考試答案記憶：AI Host → MCP Client → MCP Server（Host先、Client中間、Server後）

MCP vs RAG（Q44）⭐⭐⭐

特性 MCP RAG
核心功能 動態工具與API呼叫的整合 擴展知識庫內容
類比 AI的「手腳」（執行操作） AI的「眼睛」（查詢知識）
依賴關係 不需要向量資料庫 需要向量資料庫

Q44答案：RAG常用於擴展知識庫內容，而MCP更著重於動態工具與API呼叫的整合

🆕 A2A架構（Agent-to-Agent）⭐⭐⭐⭐⭐

核心定義

  • Google主導、已開源至Linux基金會的多智能體協作通信標準，解決「智能體孤島」問題

Client Agent vs Remote Agent（Q8）⭐⭐⭐⭐⭐

正確互動流程：Client Agent發起任務，Remote Agent執行並回傳結果

Client Agent（發起方）
  → 讀取Remote Agent的Agent Card（能力清單）
  → 發送Task Request
    → Remote Agent（執行方）接收並執行
      → 完成後回傳結果（Artifact）

❌ 常見錯誤：「Remote Agent主動分派任務給Client」（流向相反）

A2A核心概念

  • Agent Card：智能體的「名片」（能力清單，JSON格式）
  • 黑盒執行：智能體不需暴露內部實現，只需交換輸入/輸出

MCP vs A2A 最重要區別 ⭐⭐⭐⭐

協議 用途 口訣
MCP AI智能體 ↔ 外部工具/資料（工具層） AI連接工具
A2A AI智能體 ↔ AI智能體（智能體間通訊） AI連接AI

🆕 Agentic AI 智能體 ⭐⭐⭐⭐⭐

演進路徑

聊天機器人（對話）→ AI助理（輔助）→ Agentic AI（自主執行多步任務）

核心特性

  • 環境感知、維護內部狀態（跨對話記憶）
  • 自主調用工具（Tool Use / Function Calling）
  • 多步規劃與執行、自我修正能力

Solution Graph搜尋策略（Q11、Q49）⭐⭐⭐⭐

答案：使用廣度優先、深度優先或最佳優先等演算法進行探索

Q49（Solution Graph的作用）：

答案：作為代理在執行過程中的參考框架，用於組織決策步驟並支援任務推理

Agentic AI缺失工具診斷（Q14）

電商客服Agent：FAQ回答錯誤 + 無法修改訂單

缺少：向量資料庫檢索（Vector Retrieval）+ API調用（API Calling）

分析：FAQ錯誤→缺乏知識檢索；無法修改訂單→缺乏API操作能力

Agentic Workflow vs Autonomous AI Agent

比較 Agentic Workflow（工作流） Autonomous Agent（自主智能體）
路徑 預定義結構化路徑（確定性） 動態生成路徑（隨機性）
優勢 高可預測性、可控性、安全性 高靈活性、解決開放式問題

🆕 進階技術概念

聯邦學習（Federated Learning）⭐⭐⭐

定義：在不移動本地數據的前提下，透過交換模型參數來共同優化全局模型

Q2答案：希望在確保隱私前提下讓模型持續優化→ 聯邦學習（Federated Learning）

❌ 不是「同態加密」（加密運算，不同目的）；不是「零知識證明」（身份驗證用途）

運作流程：

各機構本地訓練模型（原始資料不離開）
  → 上傳模型參數/梯度（非原始資料）至中央伺服器
    → 中央伺服器聚合參數（FedAvg等演算法）
      → 下發更新後的全局模型給各機構

適用場景：醫院間疾病診斷模型、銀行間詐騙偵測、手機端個性化模型

APE（Automatic Prompt Engineer）⭐

定義：自動化提示詞工程，利用AI自動生成、評估、優化提示詞

差分隱私（Differential Privacy）

  • 加入特定噪聲，防止敏感資料外洩（隱私保護黃金標準）
  • 與聯邦學習常搭配使用

本地部署LLM（Q12）⭐

本地部署的核心優勢：確保輸入模型的敏感資料不會傳輸給第三方，提升資料隱私和自主控制

❌ 不是「零運算成本」，不是「精度更高」

即時資訊系統設計（Q29）

最佳設計：建立與外部即時行情來源同步的查詢模組（即RAG的應用）

AI系統改進優先順序（Q21）

車載語音助理：ASR錯誤率高 + LLM回覆不精確

優先提升順序：1→2→4→3（先修源頭，再修下游，最後調參數）

文本生成任務分類（Q50）

部落格續寫工具（輸入開頭→自動續寫）：生成式語言建模（Text Generation）

❌ 不是Seq2Seq（翻譯/摘要用），不是Masked LM（填空用），不是Text Classification

容易混淆的概念

1. RAG vs Fine-tuning vs Prompt Engineering

Prompt = 臨時小抄（最快、最便宜、不持久）
RAG    = 查圖書館（即時、可溯源、企業首選）
Fine-tuning = 去補習班（改造大腦、貴、學口吻不學知識）
最佳組合：能力靠Fine-tuning，知識靠RAG

2. MCP vs A2A

MCP = AI智能體 ↔ 外部工具/資料庫（工具整合協議，Anthropic主導）
A2A = AI智能體 ↔ AI智能體（跨廠商智能體通訊協議，Google主導）

3. CoT vs ToT vs Graph Prompting

CoT = 線性推理步驟（退款流程查詢）
ToT = 多分支方案比較（活動規劃）
Graph = 非線性網絡關係（社交關係、跨國合規）

4. 風險轉移 vs 風險緩解

風險轉移 = 外包/買保險 = 責任轉給第三方
風險緩解 = 採取措施降低風險發生機率或影響
風險迴避 = 不做這件事
風險接受 = 知道有風險但仍繼續（制定應急計畫）

5. 聯邦學習 vs 同態加密 vs 零知識證明

聯邦學習 = 分散式訓練模型，資料不離開本地（隱私保護+訓練場景首選）
同態加密 = 加密狀態下運算，成本極高
零知識證明 = 驗證不揭露資訊，用於身份驗證

6. Data Drift vs 過擬合（Overfitting）

Data Drift = 部署後，真實輸入資料分佈與訓練資料出現差異（時間性問題）
過擬合 = 模型訓練時對訓練數據擬合過度，無法泛化（訓練問題）

7. GAN vs VAE vs Transformer

GAN（生成對抗網路）= 圖像生成主力 = 生成式AI基礎技術（考試答案）
VAE（變分自編碼器）= 穩定的機率生成
Transformer = GPT/語言模型基礎 = 多模態生成

  • Guardrails不包括：「重建追蹤全部推理過程」

  • 個資不進模型→資料最小化+去識別化
