二、考試超出學習指引的新概念

CCChen觀察：「新法規、新趨勢、新工具，都要注意。」
煎餃Lab L122文章：考題已更新至2025年第四場，Agent/MCP為重點。

2.1 MCP (Model Context Protocol)

定義：由Anthropic主導的開放協議，讓大型語言模型（LLM）能標準化地與外部工具/資源對接，被稱為「AI的USB-C接口」。

架構流程：

用戶提問 → AI Host（如Claude Desktop）
↓
MCP Client（Host內部，維護1:1連線）
↓
MCP Server（對應一類外部能力：檔案系統/資料庫/API）
↓
執行外部操作，回傳結果給LLM
↓
LLM整合後回答用戶

三大角色：

  • MCP Host：宿主，用戶端與LLM的橋樑（如Claude Desktop、VS Code插件）
  • MCP Client：協議連接器，存在於Host內部，負責與Server維護連線
  • MCP Server：單職責服務，公開三大原語：Resources（資源）、Tools（工具）、Prompts（提示模板）

MCP vs 傳統API：MCP提供標準化、可組合、安全的通用接口；傳統各家API各自為政，需要「膠帶代碼」對接。

考試重點：

  • 記住Host→Client→Server的三層架構與職責
  • MCP解決的問題：讓AI能讀取外部工具/資料（Agent的「手腳」）
  • 社群討論：「Agentic Coding、sub-agent、skills、MCP」都可能出題
  • MCP不是讓你「會操作」，而是「知道什麼情況下用」

2.2 A2A (Agent-to-Agent)

定義：由Google主導、已開源至Linux基金會的多智能體協作通信標準，讓不同供應商/框架開發的AI Agent可以「像同事一樣」無縫協作，解決「智能體孤島」問題。

互動模式（核心流程）：

  1. Agent Discovery（智能體發現）：Client讀取Server的Agent Card（標準JSON，含名字/能力/端點）
  2. Task Request（任務請求）：Client依Card中的Skills，組織Task消息發送
  3. 執行與回報：Server通過流式(SSE)或Webhook推送進度
  4. 交付Artifact：任務完成後發回結果（文字/檔案/結構化資料）

A2A vs MCP 重要區別：

協議 用途
MCP Agent-to-Tool（AI模型連接工具/資源）
A2A Agent-to-Agent（AI智能體之間對等協作）

考試重點：

  • A2A解決跨生態、跨組織的智能體協作問題
  • Agent Card = 智能體的「名片」（能力清單）
  • 支持黑盒執行：智能體不需暴露內部實現，只需交換輸入/輸出

2.3 Agentic AI

定義：具備自主性（agency）的AI系統，能自主設定目標、分解任務、規劃執行，並根據結果自我調整。相較於傳統Chatbot的「被動回答」，Agentic AI是「主動執行者」。

Chatbot vs Agent 核心差異：

  • Chatbot：你問→它答，你是「執行小秘書」
  • Agent：承諾完成任務，自行完成「規劃→執行→檢查→修正」閉環

Agent三大核心技術條件（2024年爆發原因）：

  1. ReAct框架（Reasoning + Acting）：推理循環，「先思後行」
  2. 工具調用（Tool Use / Function Calling）：AI獲得「手腳」（讀文件/執行腳本/上網）
  3. 長上下文視窗（Long Context Window）：記憶空間擴大至20萬token

Solution Graph：

  • 將複雜問題分解為多個子任務節點，依賴關係形成有向圖
  • Agent根據圖的結構決定執行順序、調用哪些工具、維護哪些上下文狀態
  • 在Context Engineering中起關鍵作用

考試重點：

  • 煎餃Lab提到：「Agentic Coding感覺會考」
  • 重點在「判斷哪種場景適合用Agent」，而非操作
  • 人在迴圈（Human-in-the-loop）：Agent自動化執行中，什麼時候需要人介入？

2.4 Context Engineering

定義：2025年後興起的系統性方法，不再只是優化Prompt字句，而是動態、結構化地設計和管理「AI當前能訪問的全部上下文資訊」，包括歷史對話、實時數據、工具輸出、知識庫、任務狀態等。

與Prompt Engineering對比：

維度 Prompt Engineering Context Engineering
範疇 單次、靜態輸入指令 動態、系統性管理上下文
主要用途 單步問答、短對話 多步推理、長時任務、複雜Agent
核心挑戰 詞句優化、意圖表達 資訊過載（context rot）、有效篩選
典型失效 prompt不精確→輸出偏差 上下文衰減→丟失細節

為何Context Engineering取代Prompt Engineering：

  • 生產級AI Agent需要動態、結構化上下文，而非一條條prompt拼湊
  • Anthropic研究：有效的Context Engineering可將AI錯誤率降低40~54%
  • Gartner認為Context Engineering是未來AI應用的關鍵能力

考試重點：

  • 兩者是遞進關係：Prompt Engineering是Context Engineering的「低維切面」
  • 記住核心差異：「如何問問題」vs「如何為AI持續提供最合適的知識和狀態」

2.5 聯邦學習 (Federated Learning)

定義：分散式機器學習方法，各資料擁有者（手機、醫院、銀行）在本地訓練模型，只上傳模型參數（非原始資料）至中央伺服器聚合，原始資料始終留在本地。

架構：

本地裝置/機構 → 在本地訓練 → 僅上傳模型參數/梯度
↓
中央伺服器聚合（FedAvg等演算法）
↓
更新全域模型，下發各裝置

適用場景：

  • 醫療：多家醫院聯合訓練疾病預測模型，病患資料不外流
  • 金融：多家銀行聯合做信用評分，商業機密不洩露
  • 終端設備：手機輸入法個人化（Google Gboard的實際應用案例）
  • 政府機構：跨機關資料協作，符合個資法規

隱私保護機制：

  • 本地訓練，不上傳原始資料
  • 差分隱私（Differential Privacy）：加入噪聲防止推斷個人資訊
  • 同態加密（Homomorphic Encryption）：加密狀態下聚合

考試重點：

  • 社群提到：「最愛考聯邦學習」
  • 核心概念：解決「資料協作」vs「隱私保護」的矛盾
  • 適用條件：資料不能集中 但 需要共同訓練 的場景

2.6 RAG vs Fine-tuning 選擇標準

RAG（檢索增強生成）：不改動模型，查詢外部知識庫後附加到Prompt。比喻：「考試可以查資料」。

Fine-tuning（模型微調）：用領域資料再訓練模型，知識融入權重。比喻：「把知識背起來」。

選擇標準速查：

需求 選RAG 選Fine-tuning
資料需頻繁更新 ✅ ❌
需要即時最新資訊 ✅ ❌
格式/語氣需標準化 ❌ ✅
合規/資料不可進模型 ✅ ❌
計算成本考量 ✅成本低 ❌資源需求高
領域術語/邏輯固定 ❌ ✅

口訣：「資料動，選RAG；格式定，選Fine-tuning；務實混合才高效」

考試重點：

  • 最大本質差異：RAG不更改模型權重，Fine-tuning會調整權重
  • 若頻繁更新內容採用Fine-tuning→資料過期且成本高（常見陷阱題）
  • 兩者可結合：「能力（格式邏輯）」靠Fine-tuning，「知識（最新資料）」靠RAG

三、AI治理法規速查

EU AI Act（歐盟AI法）

四大風險分類（由高到低）

風險等級 說明 例子
不可接受風險（全面禁止） 對歐洲價值、民主、人權構成極端危害 社會評分系統、潛意識操控、即時公共場所生物辨識（警用）、推斷情緒的職場AI
高風險（嚴格監管） 涉及個人健康、安全、基本權利 醫療診斷AI、人臉辨識入境、自動駕駛、法庭量刑AI
有限風險（透明度義務） 影響較小，但需告知用戶與AI互動 聊天機器人（客服）、Deepfake生成
最低風險（無需額外監管） 日常低影響應用 垃圾郵件過濾、推薦系統、遊戲NPC

考試高頻考點：

  • 違規罰款：2000萬歐元或4%全球年營業額（取最高）
  • 歐盟AI法四大倫理原則：公平、可解釋性、尊重人類自主權（注意：透明性❌不是四原則之一，常出陷阱題）
  • GDPR資料洩漏通報：企業必須在72小時內通知監管機關

台灣《人工智慧基本法》

2025年12月23日三讀通過，主管機關：國家科學及技術委員會

七大原則：

  1. 永續發展與社會福祉：AI發展須符合社會長遠利益
  2. 人類自主：維護人類自主決策權
  3. 隱私保護與資料治理：強化個資保護與合理利用
  4. 資安與系統安全：提高AI系統穩健性
  5. 透明與可解釋性：AI決策需提供合理資訊揭露
  6. 公平與不歧視：防止AI偏見與歧視
  7. 問責：明確責任歸屬與監督追責

考試重點：

  • CCChen強調：「台灣人工智慧基本法與ISO 42001 AIMS，是今年最重要主題」
  • 社群記錄：約考3~5題基本核心概念與應用
  • 核心精神：「以人為本」
  • 行政院將成立「國家人工智慧戰略特別委員會」
  • 高風險AI需設責任停損標示及風險分級管理

四、CCChen備考指南摘要

來源：https://vocus.cc/article/695f29dcfd89780001822622（2026/1/8更新）

三大出題趨勢（2026年）

趨勢一：AI基礎概論從「定義題」→「判斷題」

  • 不考「什麼是非監督式學習」，考「這個商業場景為何用非監督式更合理？」
  • 準備方式：每個概念都要能回答「用來解決什麼問題」

趨勢二：生成式AI考「規劃邏輯」，不考「工具介紹」

  • 三個考層次：
  • 生成式AI的角色定位（適合輔助什麼、不適合取代什麼）
  • 生成式AI×企業流程（為何不能全自動、為何要Human-in-loop）
  • 生成式AI的風險（幻覺、隱私、上下文錯置）

趨勢三：No Code/Low Code、Agent、MCP——知道「怎麼選」

  • 考的不是「操作熟不熟」
  • 考的是「什麼時候用、什麼時候不適合用」

題型分佈（約略）

  • 情境判斷題：40~45%（主戰場）
  • 單一概念理解題：25~30%
  • 觀念比較題：20%
  • 制度/治理/倫理：5~10%（比例不高但容易失分）

CCChen的5大備考建議

  1. 不問「會不會考這個」，而問「它解決什麼問題」
  2. 情境題要練「為什麼不是其他選項」（逐一排除）
  3. 生成式AI不背功能，背「限制」
  4. 用白話重新寫一次概念（寫不出來代表還沒懂）
  5. 最後兩週不補新東西，做「概念整合」
